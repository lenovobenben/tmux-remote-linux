# tmux-remote-linux 唯一提示符边界方案

## 背景

当前 `run.sh` 使用 `BEGIN/END marker` 截取命令输出：

1. 本地生成 `__CODEX_RUN_<timestamp>_<pid>_BEGIN__` 和 `__CODEX_RUN_<timestamp>_<pid>_END__`。
2. 把用户命令 base64 包装成远端 `bash` 子进程。
3. 远端先打印 BEGIN，执行命令，再打印 `END:<exit_code>`。
4. 本地通过 `tmux capture-pane` 在 scrollback 中查找 BEGIN/END，截取两者之间的内容。
5. `read.sh` 需要额外过滤 wrapper 命令和 marker 噪声。

这个模型的问题是：命令输出、wrapper 输出和控制 marker 混在同一个终端 scrollback 中。输出很长、命令被中断、END 被滚出 history、或上一次 marker 残留时，`run.sh` 会频繁进入 `stale run marker detected` 或 `end marker not found yet`，需要 `read.sh` 和 `tmux clear-history` 人工恢复，效率低且容易误判。

## 目标

用唯一 shell prompt 作为 BEGIN/END marker 的恢复辅助，而不是直接替代 BEGIN/END marker。

目标效果：

- BEGIN/END marker 仍然作为主边界，继续负责输出截取和退出码恢复。
- 当 BEGIN/END marker 缺失或陈旧时，用“唯一 prompt 再次出现”判断 shell 是否已经回到空闲。
- Ctrl+C、中断或 marker 丢失后，可以通过 prompt 判断 shell 是否已经回到空闲。
- 减少 `stale marker` 和反复清 tmux history。
- 保留现有生产确认、日志、输出截断、REPL 拒绝等安全能力。

## 实施调整

第一版实现不直接替代 BEGIN/END marker。最终采用更保守的混合策略：

- BEGIN/END marker 仍然是 `run.sh` 的主边界，用于截取输出和获取退出码。
- 受管 prompt 只作为恢复辅助，用于判断 marker 缺失或陈旧时 shell 是否已经回到空闲。
- 受管 prompt 在环境确认和生产确认之后初始化，之后复用当前 shell 会话中的同一个 prompt。
- 如果旧 BEGIN marker 后面已经出现过受管 prompt，说明 shell 后来回到过空闲，`run.sh` 不再把该旧 marker 当作仍在运行的命令。
- 如果 pane 通过人工 SSH、`tmux-oasis-cli` 等方式进入了另一个 Linux shell，下一次 `run.sh` 发现当前可见 prompt 不是受管 prompt 时，会在新 shell 中重新安装。
- 启用 prompt guard 后，托管 pane 中的 `PS1` 和 `PROMPT_COMMAND` 属于这个 skill 的状态，用户不应手动修改；需要自定义 prompt 时应关闭 `REMOTE_TMUX_PROMPT_GUARD`。

受管 prompt 固定为：

```text
__31D763DA06_TRL_<counter>_<status>__
```

其中 `31D763DA06` 是自然常数 `e` 的十六进制小数点后第 91-100 位，只作为固定 magic prefix，不是安全 token。

## 方案概述

进入目标 Linux shell 后，临时设置一个只在当前 shell 会话生效的唯一 `PS1`，不修改远端配置文件。

最终 prompt 形态：

```text
__31D763DA06_TRL_<counter>_<status>__
```

第一版保留一个可见且低误匹配概率的 token，优先保证可调试性。

执行流程：

1. 初始化阶段确认当前 pane 是普通 Linux shell，不是 MySQL、Python、Spark shell、Oasis、TUI 等。
2. 临时设置 `PS1` 为唯一 prompt，并调大 tmux history limit。
3. 每次 `run.sh` 执行前记录当前 prompt 出现位置或 capture 快照。
4. 发送仍包含 BEGIN/END marker 的原有 wrapper。
5. 优先用 BEGIN/END marker 截取输出和解析 exit code。
6. 如果 marker 缺失或陈旧，再检查唯一 prompt 是否已经再次出现。
7. prompt 只判断 shell 是否空闲；exit code 仍以 END marker 中的状态为准。

## exit code 设计

不能只靠 prompt 出现判断成功，因为命令失败也会回到 prompt。

推荐方式是在 `PROMPT_COMMAND` 中记录上一条 shell 状态，并把状态放进 prompt token。第一版只把这个状态作为诊断信息，`run.sh` 的命令退出码仍以 END marker 为准。

示例思路：

```bash
export __TRL_COUNTER=0
PROMPT_COMMAND='__trl_status=$?; __TRL_COUNTER=$((${__TRL_COUNTER:-0} + 1)); PS1="__31D763DA06_TRL_${__TRL_COUNTER}_${__trl_status}__ "'
```

本地等待并解析：

```text
__31D763DA06_TRL_<counter>_<status>__
```

注意：`PROMPT_COMMAND` 和 `PS1` 只适用于 bash；zsh 需要使用 `precmd` 或 zsh 兼容逻辑。第一版可以明确只支持 bash，检测到 zsh 时走兼容分支或拒绝。

## 当前脚本影响

### `run.sh`

需要重写核心边界逻辑：

- 保留 `begin/end` 生成和 stale marker 检测。
- 保留远端 `printf BEGIN`、`printf END:<status>` 包装。
- 增加 prompt 初始化和确认逻辑。
- 发送命令前记录当前 prompt counter。
- 仅在 marker 缺失或陈旧时读取 prompt counter 判断 shell 是否空闲。
- exit code 仍从 END marker 解析；prompt token 中的 status 不作为命令结果来源。

仍应保留：

- `REMOTE_TMUX_ENV` 校验。
- 生产环境审批。
- REPL prompt 拒绝。
- 输出行数和字节数限制。
- JSONL 审计日志。
- pending/timeout 状态。

### `read.sh`

可以简化：

- 不再过滤 `__CODEX_RUN...BEGIN/END__`。
- 仍可过滤 prompt 初始化 wrapper。
- 可选隐藏唯一 prompt，避免用户看到大量机器 token。

### `send.sh`

`send.sh` 仍然只负责发送命令，不等待完成。

可选改进：

- 如果唯一 prompt 未初始化，提示用户先用 `run.sh` 或新增初始化脚本初始化。
- 或保持现状，避免扩大改动面。

## 初始化策略

推荐新增内部函数或脚本：

```text
ensure_prompt_initialized(target)
```

职责：

- 读取最近 pane 内容，判断是否已经存在当前 session 的唯一 prompt。
- 如果没有，确认当前是普通 shell prompt。
- 发送 `export PS1=...` / `PROMPT_COMMAND=...`。
- 等待唯一 prompt 出现。
- 设置或建议设置 tmux history limit，例如：

```bash
tmux set-option -t "$session" history-limit 200000
```

history limit 是本地 tmux 设置，不影响远端系统配置。

## 输出截取

基本截取规则：

1. 发送命令前，捕获当前 pane。
2. 发送命令后，等待新的唯一 prompt。
3. 在 capture 内容中找到“发送命令后的第一个命令 echo 行”和“下一次唯一 prompt”。
4. 输出为两者之间的内容。

如果 shell 不 echo 命令，或命令跨行、被终端自动换行，不能完全依赖命令 echo。更稳的方式是维护 prompt 计数器：

```text
prompt #17
发送命令
等待 prompt #18
截取 prompt #17 之后到 prompt #18 之前
```

这要求 prompt 中包含递增 counter。counter 可在 `PROMPT_COMMAND` 中递增。

## 超时与 Ctrl+C

超时：

- 如果没有看到下一次唯一 prompt，返回 pending，不伪造 exit code。
- 输出最近若干行，说明命令可能仍在运行。

Ctrl+C：

- 用户或 agent 发送 Ctrl+C 后，正常 shell 会重新打印 prompt。
- 只要 prompt token 出现，即可闭合边界。
- exit code 通常为 `130`，可从 prompt 状态中解析。

## 边界与风险

这个方案只适用于普通 shell。

必须继续拒绝：

- `mysql>` / `MariaDB>`
- `python >>>` / `...`
- `spark-sql>` / `scala>`
- Redis、psql、Oasis `work>`
- `vim`、`less`、`top`、`watch` 等 TUI
- 嵌套 SSH 过程中未回到稳定 shell prompt 的状态

潜在风险：

- 用户手动修改了 PS1/PROMPT_COMMAND。
- 远端环境的站点 profile、审计 hook、终端标题集成、kube/venv/git prompt 辅助逻辑，或用户工作流依赖自定义 `PROMPT_COMMAND`。
- 远端 shell 不是 bash/zsh。
- 用户或其他 adapter 切换到新主机后，还没回到普通 shell prompt 就调用 `run.sh`。
- 命令本身修改 PS1、PROMPT_COMMAND、stty、terminal mode。
- 输出极大，prompt 边界仍可能滚出 tmux history。
- 程序输出伪造了 prompt token。固定 magic prefix `31D763DA06` + counter 可以把误匹配概率降到很低。

因此第一版仍要保留输出截断和 timeout，不要把 prompt 边界当作无限可靠传输。

## 大输出兜底

唯一 prompt guard 能辅助判断 shell 是否回到空闲，但不解决无限大输出问题。

建议保留现有“不要 dump 大输出”的使用原则，并为后续版本预留远端文件兜底：

```text
短输出：BEGIN/END marker 截取，prompt guard 辅助恢复
大输出：远端临时文件 + bounded tail/read
```

第一版只实现 prompt guard，不立即做远端文件传输。

## 迁移步骤建议

1. 在 `run.sh` 中新增 prompt guard 开关：

```bash
REMOTE_TMUX_PROMPT_GUARD=1
# 设置为 0 可以回退到旧的纯 marker 行为
```

2. 实现 prompt 初始化和 counter 判断函数。
3. 在非生产环境和本地 mock tmux session 中测试。
4. 保留 BEGIN/END marker 主边界，只在 marker 缺失或陈旧时使用 prompt guard 判断 shell 是否空闲。
5. 不删除 marker 相关逻辑。
6. 更新 `SKILL.md`、`README.md`、reference 文档。

## 验收用例

至少覆盖：

- `echo hello`
- `false`
- `sleep 2; echo done`
- `printf` 输出不带换行
- 输出 1000 行
- 命令输出包含旧的 `__CODEX_RUN...` 字符串
- 命令输出包含类似 prompt 的普通文本
- `Ctrl+C` 中断 `sleep 100`
- 进入 `mysql>` 后 `run.sh` 仍拒绝执行
- zsh 环境或非 bash 环境的行为
- 用户手动输入干扰后的恢复提示

## 结论

唯一 prompt 边界方案可行，且比 BEGIN/END marker 更适合 tmux 交互模型。

推荐落地原则：

```text
唯一 prompt 负责命令边界
exit code 进入 prompt 状态
tmux history 负责短输出
远端文件作为后续大输出兜底
```

这能显著降低 stale marker、history 清理和重复执行问题，同时保持 `tmux-remote-linux` 当前的安全边界。
